Handoff Flows

Setter → Closer
NEW → Warm → Hot → Link Sent → Call Booked ⤳ Closer takes call → Won or FU Immediate / Nurture or Dead
Setter sees FU/Nurture as greyed-out columns (read-only). Can coordinate with closer but cannot move the card.
Dialer → Closer
New (Not Contacted) → Called, No Answer → Call Booked ⤳ Closer takes call → Closed or No Close → FU or Not Interested
Dialer sees FU as greyed-out (read-only). No Show stays in dialer kanban — dialer handles recovery calls.
No Show Recovery
No Show → Setter or Dialer follows up (whoever booked the original call) → Re-booked or Lost

Rules

Rule Why
Bottom bar = definitive only Won and Lost/Not Interested/Dead are terminal. Everything that can be followed up stays in the kanban columns.
Greyed columns = read-only visibility After handoff (BOOKED), the setter/dialer can see where their lead is in the closer pipeline, but cannot move it. Coordinate via Slack if needed.
One owner per stage Each lead_pipeline row has an owner_role. The role that owns the row controls the card. UNIQUE(lead_id, owner_name, owner_role) prevents duplicates.
Handoff creates a new row When a setter/dialer books a call, the system creates a closer pipeline row. Both rows coexist — the setter/dialer row stays at BOOKED (greyed), the closer row is active.
No Show = kanban, not bottom bar No-shows need recovery calls. They stay in the active kanban where the responsible role (setter or dialer) can follow up.
