Key: Setter NEW = Dialer NEW

Both setter and dialer "New" columns read from the same leads table. The only difference is the source filter:

Setter Typeform, Meta Lead Ads, ManyChat
Dialer CSV imports (LB90 buyers, partial apps, Meta ads list)

Handoff Flows

Setter → Closer
NEW → Warm → Hot → Link Sent → Call Booked ⇒ Closer takes call → Won or FU / Nurture or Dead
After Call Booked, setter sees FU/Nurture as greyed-out columns. Can coordinate with closer via Slack but cannot move the card.
Dialer → Closer
New (CSV import) → Called, No Answer → Call Booked ⇒ Closer takes call → Closed or No Close → FU or Not Interested
Dialer sees FU as greyed-out (read-only). No Show stays in dialer kanban for recovery calls.
No Show Recovery
No Show → Setter or Dialer follows up (whoever booked the original call) → Re-booked or Lost

Rules

RuleWhy
Bottom bar = definitive only Won and Lost/Not Interested/Dead are terminal. Everything that can be followed up stays in the kanban columns.
Greyed columns = read-only After handoff (Call Booked), the setter/dialer can see where their lead is in the closer pipeline, but cannot move the card.
One owner per role per lead Each lead_pipeline row has an owner_role. UNIQUE(lead_id, owner_name, owner_role) prevents duplicates. Multiple roles can have rows for the same lead.
Handoff creates a closer row When a setter/dialer books a call, the system creates a closer pipeline row. Both rows coexist — setter/dialer at BOOKED (greyed), closer active.
No Show = kanban, not bottom bar No-shows need recovery calls. They stay in the active kanban where the setter or dialer can follow up.
