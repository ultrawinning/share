3 · Manual vs Automated — who does what

Everything a human must do for the journey to advance, vs what runs hands-free. The journey's middle (confirm / cancel / no-show / outcome) and its end (onboarding chase, coach assignment) are human-dependent.

MANUAL A human must…

  1. Star the Slack booking message (setter attribution trigger)
  2. Claim leads onto setter kanban cards (nothing auto-creates them)
  3. Run all imports outside webhooks (GHL import, dialer import, CSV)
  4. Confirm calls (dialer loop; actor self-reported)
  5. Record EVERY durable cancellation — the automated lane writes nothing
  6. Work dialer reschedule lanes
  7. Record no-shows (closer outcome submit only)
  8. Submit won / lost / follow-up (the cascade is automated after the click)
  9. Claim closer identity at outcome (or manual override)
  10. Drag cancel/no-show cards into recovery on the dialer board
  11. Assign the coach (click in /coaching/roster)
  12. Execute the fulfilment handoff itself (Bea/Sheena acting on a Slack message — no task record)

AUTO The machine does…

  1. Lead ingestion (Typeform / Meta / GHL webhooks → identity, lead, touchpoint, GHL mirror)

  4. Setter attribution sync (emoji cron + UTM backfill)
  5. Payment bookkeeping (payment row, lead close, EOD write-back, CAPI, Slack win)

Glossary (the few unavoidable terms)

GHL (GoHighLevel)
The external CRM/calendar product. Booking, closer assignment, and onboarding workflows live there, outside this codebase.
Webhook
An automated message an external system (Typeform, Meta, GHL, Stripe…) sends to HPFHQ the moment something happens. "Fail-open" = the receiving door accepts messages even when its password is missing.
Cron
A scheduled job that runs automatically (e.g. every 30 minutes / daily at 08:00).
lead_pipeline ("card")
The single database table holding every kanban card on the setter, dialer, and closer boards — one card per (lead, owner, role).
lead_id
The lead's unique database ID — the join key that connects a lead's cards, close records, and payments.
hpf_id
The person's identity-graph ID, linking emails, phones, GHL contact, Stripe customer, and Trainerize client to one human.
EOD entry (closer_eod_entries)
The closer's end-of-call outcome record — the "close claim" (vs. the payment tables, which are the "close proof").
A7a flip
The automated write-back: when a closer marks won/dead, the same lead's setter/dialer cards get stamped WON / LOST_NO_FU.
§7 invariant
The internal write rule: every card update must be keyed by lead_id and scoped to the writer's role, so one board can never corrupt another's cards.
Dry-run / feature flag
Code that executes but is switched off — it logs what it WOULD have done (e.g. GHL reflection: 317 suppressed sends in 14 days) without doing it.
1,000-row cap
The database API silently returns at most 1,000 rows for an unbounded query — no error, just missing data.
