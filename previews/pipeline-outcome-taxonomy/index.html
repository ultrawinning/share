Master Pipeline — Outcome Taxonomy CANON DRAFT · 2026-06-27

A lead lives on two orthogonal axes plus a board working layer. AXIS-1 = how far down the funnel it got (a monotonic 5-rung ladder — the only thing HPFHQ counts). AXIS-2 = what's happening / what went wrong off that highest rung (the disposition state machine). AXIS-L2 = each board's private sub-stages, every one nesting under one milestone. There is deliberately no "Lost" milestone — a loss is a disposition recorded at the rung the lead reached.

AXIS-1 · MILESTONE

How far did it get?

leads.current_stage — monotonic lead → qualified → booked → showed → closed. Only advances; never holds a disposition. The funnel HPFHQ counts.

AXIS-2 · DISPOSITION

What's happening / what went wrong?

leads.call_outcome — the mutable state off the highest rung (NULL=active). A state machine: active / transient / working / holding / terminal.

AXIS-L2 · SUB-STAGE

The board's working columns

lead_pipeline.(owner_role, stage) — unlimited per board; every one rolls up to exactly one milestone. Never counted directly.

⚠ Saving invariant: all three axes are the spine — they drive boards + lifecycle and are never a metric / comp / credit source. DNC (do_not_contact) is a 4th, orthogonal compliance flag — a hard outbound block, not a pipeline position.

The "Lost" umbrella — a reporting group, never a milestone: ghosted + not_interested + pre_call_dq + quick_dq (+ DNC). Recorded at the milestone the lead reached, so you can always ask "of the leads that booked, how many did we lose and why." Ghosted (silence through 3 FUs) ≠ Not-Interested (a spoken "F no") — they behave differently downstream. Both replace the retired generic dead (0 live rows — unreachable by any writer).
🔒 DNC — the compliance flag (orthogonal, NOT a disposition). A hard block on all outbound, basis = US TCPA Opt-Out Rule. Live state: the store columns exist (do_not_contact / sms_dnd) but the send-time gate is not wired (checkSmsDnd has zero callers) — a load-bearing build item. Re-engagement: a DNC lead reaching back out raises a leadership-cleared flagged review (logged reason), never an auto-reopen.

3 · The outcome-per-milestone validity matrix

Which disposition is valid at which milestone — grounded in the live DB census. ✅ valid/expected · ⚠️ valid-edge or to-build · ✗ invalid (a writer producing it = a bug / drift) · — n/a. Cells marked ✗ with a live count are drift to flag.

Reading it: the closed column is NULL only — the 18 live closed × disposition rows are drift (a payment-webhook close didn't clear the prior disposition) and are currently unmonitored (the drift checker suppresses closed leads). Transients live at booked; no_close /quick_dq require an attended call (showed); pre_call_dq is pre-attendance; not_interested/ghosted can land anywhere a lead is lost after engagement.

Adding a stage = add a sub-stage to the map under one of the five milestones (in code — no self-serve editor); a coverage guard asserts every live stage is mapped, so an unmapped/half-defined sub-stage is impossible. The 5 milestones + the disposition enum are canon — changing them is a deliberate, gated change. Full reference: docs/plans/2026-06-27-master-pipeline-outcome-taxonomy-canon.md.
