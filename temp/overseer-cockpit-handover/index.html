Overseer + HPFHQ Cockpit Alerting — Handover

From  Emil (HPFHQ owner) — 2026-05-20
For  Overseer's owner / maintainer
Context  HPFHQ cockpit consolidation Phase 0.5 — design lock around number-correctness + failure surfacing
Subject  Overseer is the alerting surface for HPFHQ cockpit failures. Here's what Overseer needs to do for the cockpits and how the data flows.

§1The decision

HPFHQ's marketing + sales cockpits (with setter/dialer/closer workstations to follow) are being consolidated as the canonical operational surfaces. As part of that, we locked a number-correctness contract: when cockpit data is stale, partially-fetched, or a data source has errored, the cockpit shows the last-known value with an "as of [timestamp]" updated to surface the staleness. The cockpit does not hide problems silently.

Overseer is the surface where failures are noticed and routed. Not Telegram, not Slack-direct, not console logs. Overseer is the dashboard the team checks when something looks wrong, and Overseer is where alerting routes converge before fanning out to wherever they need to go (Slack channel, on-call person, etc).

Important context: The HPFHQ codebase has historically used lib/telegram.ts:reportFailure() as the "alert this failed" helper. The function actually posts to /api/triage/intake (not directly to Telegram), but the misleading naming has created a pattern where docs + plans reference "Telegram alerts" when describing the alerting topology. We're cleaning that up — a Codex audit is enumerating every Telegram-as-alerting reference in the HPFHQ code/docs for replacement. The cleanup PR will land separately from this handover.

§2What Overseer needs to do for HPFHQ cockpits

Four monitoring tasks. All passive — Overseer reads from existing HPFHQ Supabase tables + does HTTP health-checks. No writes to HPFHQ data.

2.1 — Health-check the 5 live surfaces

HTTP GET against each surface every N minutes (suggest 5min during business hours, 30min off-hours). 200 status = green. Anything else = surface health alert.

Recommend exposing simple /api/health/cockpit + /api/health/workstation JSON endpoints in HPFHQ that return {status: "ok" | "degraded" | "down", checks: {...}}, gated by a service token. Overseer hits the JSON endpoints rather than scraping HTML. We can build those when Overseer is ready to consume them — flag and I'll spec.

2.2 — Cron failure detection

HPFHQ has ~40 scheduled crons in Vercel. The cockpits depend on ~5 of them:

CronScheduleWhat breaks if it fails
ad-snapshotDaily 06:00 UTCMarketing cockpit ad spend stale; per-campaign breakdown stale
metric-snapshotsDaily 07:30 UTCMorning Prep, scorecard, hub key-metrics, daily-summary, Nova vocabulary all stale
sync-setter-reactionsEvery 30minSetter workstation Slack reaction sync stale
missing-fields-alertDaily 14:05 UTCLead-data quality alerts delayed
showed-classificationDailySales cockpit showed/no-show classification drift

Detection mechanism: HPFHQ writes cron run records to a cron_runs table (already exists). Each row has cron_name, started_at, finished_at, status, error_message. Overseer polls this table: any cron without a successful run within (expected_interval × 1.5) = failure surface. Specific query shape:

SELECT cron_name, MAX(finished_at) AS last_success
FROM cron_runs
WHERE status = 'ok'
GROUP BY cron_name;
-- compare each last_success against expected_interval per cron_name

HPFHQ side TBD: expose /api/health/crons JSON if Overseer prefers polling that vs the table directly. Flag your preference.

2.3 — Webhook receipt gap detection

HPFHQ logs every webhook receipt to a webhook_receipts table via logWebhookReceipt() in webhook routes. Sources: GHL, Fanbasis, Stripe, Typeform, Meta, ManyChat.

Detection mechanism: Overseer polls receipts grouped by source. During business hours (e.g. Mon–Sat 09:00–22:00 local time), if any of {GHL, Fanbasis, Stripe, Typeform} has zero receipts in the last 2 hours, surface a gap alert. Off-hours threshold = 6 hours.

Why this matters: in 2026-04 we had a 3-day silent failure where Fanbasis webhook env vars were set wrong with a trailing whitespace — Brandon noticed because closer EOD digests stopped firing. Webhook gap detection from Overseer side would have caught it in ≤2h.

2.4 — "As of" indicator data source

The cockpit shows "Ads: 6h ago · EOD: live · Payments: live" indicators per data source. These read from the same data Overseer is polling — likely makes sense for the cockpit to query a single endpoint that Overseer also writes to (or for both to read the same source-freshness materialized view).

Suggested shape: data_source_freshness view with columns (source, last_write_at, status). Cockpit reads for the indicator; Overseer reads for the dashboard tile.

§3Where this lives in Overseer's dashboard

Suggested layout — one section per HPFHQ subsystem:

  • HPFHQ → Surfaces — 6 tiles (marketing, sales, settings cockpits + closer/setter/dialer workstations). Green/red per HTTP health.
  • HPFHQ → Crons — table of cron_name + last_success + status. Sorted by most-overdue.
  • HPFHQ → Webhooks — per-source last receipt + status. Same shape as the cron table.
  • HPFHQ → Data freshness — per-source last write timestamp; mirrors what the cockpit "as of" indicators show.

Each section's tile color + alert routing inherits Overseer's existing conventions. No new alerting infrastructure on HPFHQ side beyond the existing tables/views.

§4Integration shape — HPFHQ ↔ Overseer

Read-only by default. Overseer pulls from HPFHQ — HPFHQ does not push to Overseer. Lower coupling, no new HPFHQ outbound integrations to maintain.

Two options for how Overseer reads:

OptionMechanismPros / Cons
A — Direct SupabaseOverseer has its own service-role key for the HPFHQ Supabase project; queries cron_runs, webhook_receipts, data_source_freshness directly.Pros: minimal HPFHQ work, no new endpoints. Cons: Overseer carries HPFHQ secrets; schema changes in HPFHQ can break Overseer silently.
B — HTTP JSON endpointsHPFHQ exposes /api/health/crons, /api/health/webhooks, /api/health/data-freshness gated by a service token; Overseer polls these.Pros: HPFHQ owns the contract; schema changes hidden behind the JSON. Cons: ~3 new endpoints to maintain.

Recommend B — slightly more HPFHQ-side work but cleaner contract. Endpoints are thin (SQL → JSON). Flag your preference and I'll spec.

§5What I need from you

  1. Confirm or push back on the four monitoring tasks (§2.1–2.4) — does Overseer have capacity for these? Anything missing?
  2. Pick Option A or B (§4) for the integration shape.
  3. Confirm Overseer dashboard layout (§3) — does an "HPFHQ" parent section with 4 sub-tiles fit the existing Overseer UI, or do you organize differently?
  4. Alerting routes — when something surfaces in the HPFHQ section, where does Overseer route the alert (Slack channel, on-call, email)? Not for me to dictate, just need to know so the HPFHQ cockpit handover docs can describe the loop end-to-end.
  5. Timing — HPFHQ Phase 6 of the cockpit consolidation plan is when this lands. Sequence: HPFHQ exposes the endpoints/tables → Overseer wires them up → smoke test. ETA TBD per Overseer's own roadmap.
What you don't need to worry about: the cockpit canonical layer rework, the dictionary surface, the Sheet, the goals layer. Those are HPFHQ-internal work. Your scope is just the failure surfacing.
