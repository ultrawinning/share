Problem

HPFHQ has extensive documentation — integration guides, architecture specs, context docs, manifests — but it doesn't get loaded into the right Claude session at the right time. When working on the closer workstation, Pathfinder could find knowledge articles about sales but couldn't recommend "read components/closer/KanbanPipeline.tsx." Context exists but gets ignored.

What It Does

Semantic Code Search

Every .ts, .tsx, .md, and .sql file is embedded with Voyage-3-lite (512 dimensions) and stored in a code_embeddings Supabase table. When you query Pathfinder, it returns the most semantically similar files to your task description.

Example: querying "closer kanban pipeline" returns KanbanPipeline.tsx (1.0), UnifiedPipeline.tsx (0.92), SetterKanban.tsx (0.86), UnifiedLeadCard.tsx (0.80) — all genuinely relevant.

Context Packs

Pre-computed file sets for known task types. When the CC hook detects what you're working on (via prompt keywords), it looks up the matching pack and tells Claude which files to read before starting.

CC Hook Enforcement

The launch packet now includes an INSTRUCTION directive that forces Claude to read mandatory files before starting work. This isn't a suggestion — it's injected as a system instruction on every prompt.

INSTRUCTION: BEFORE starting any work, you MUST read the following
files using the Read tool. Do not begin implementation, analysis,
or suggestions until you have read each one.

**Context Pack** (build-workstation/sales):
- [ ] READ: state/SALES.md
- [ ] READ: docs/PRD_Closer_Workstation.md
- [ ] READ: docs/Workstation_Design_System.md
**Also read if relevant:** app/closer/, components/closer/
**Semantically related:** KanbanPipeline.tsx, UnifiedPipeline.tsx

Codex Synthesis (Layer 2)

OpenAI Codex (gpt-5.5, subscription) reads each indexed file and generates:

  • Summary: one-line description of what the file does (max 120 chars)
  • Relationships: array of file paths it imports or references

These populate summary and relationships columns on code_embeddings, enriching the semantic search results with human-readable context.

Key Technical Details

  • Embedding model: Voyage-3-lite, 512 dimensions (same as existing HIVE tables)
  • Tables: code_embeddings (file vectors + metadata), context_packs (pre-computed read sets)
  • RPC: match_code_embeddings(query_embedding, match_count, filter_project, filter_domain)
  • Indexer: scripts/index-code-embeddings.ts — hash-based staleness detection, cross-project support via --project flag
  • Safeties: MAX_EMBEDS_PER_RUN=500, 5-min Codex timeout, 50-file synthesis batch cap
  • Cross-project: HPF (1,171 files) + Overmind (75 files) indexed. Single table with project column for isolation.
  • Nightly cron: launchd at 03:00 — re-indexes both repos, runs Codex synthesis on next 50 unsummarized files

For White-Label / Replication

This is an Overmind tool, not HPF-specific. Any project can be indexed with:
npm run index:code -- --project <name> --root <path>

The architecture is designed to be portable:
  • Single code_embeddings table serves all projects (project column)
  • Context packs are per-project, per-domain, per-task-type
  • CC hook scoping via CWD guard (only fires in matching project directory)
  • Codex synthesis is generic — reads any codebase, generates summaries
For white-label HQ: index the customer's repo, seed their context packs, done.
