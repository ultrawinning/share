§1Decisions resolved by UX research

D1Tooltip pattern locked research #

Decision: Click-to-open (i) icon after every metric label. Opens a popover (200–320px) anchored to the icon. Contains four required fields:

  1. One-sentence plain-English definition
  2. Formula or source logic ("Sum of leads where booked=true, bucketed by typeform_submitted_at")
  3. "As of [timestamp]" — when the metric was last computed
  4. "Not the same as [other metric] because [reason]" — for any metric that has a near-namesake on another cockpit

Bottom of popover links to the full dictionary tab. Mobile: tap = bottom sheet (slides up 40% of viewport).

Source: Amplitude Amplitude's (i) icon pattern is best-in-class. Hover-only tooltips fail on touch, fail on dense grids, and don't hold the 80–150 words a definition needs. Both lessons-locked (2026-04-07) and the research converge here. The v2 plan's original Radix hover proposal is wrong — overruled.

HPFHQ already has a working reference: components/dashboard/DashboardMetricCard.tsx:255-277 implements TooltipButton click-toggle. Copy that pattern into the cockpit primitive.

Research source: Amplitude out-of-the-box metrics; ChartMogul MRR/ARR/Churn popovers; Datadog Metrics Summary.

D2Period / date controls locked research #

Decision: Looker Studio pattern. Single date range control widget in the cockpit header. Contents:

  • Preset list grouped by category with explicit labels and visual separators:
    • Rolling: Today, Yesterday, Last 7 days, Last 28 days, Last 30 days, Last 90 days
    • Calendar Week: This week (starts Monday), Last week
    • Calendar Month: This month, Last month
    • Calendar Quarter/Year: This quarter, Last quarter, This year, Last year
    • Custom
  • Custom = calendar grid (click start, click end), NOT two text inputs. Text display of selected range above the grid for confirmation.
  • Comparison period is opt-in, disabled by default. When enabled, default = "Previous period" (same length, immediately preceding). Other options: same period last year, custom.
  • "Data as of [timestamp]" indicator near the date control. Not buried in a settings page.
  • Date state in URL. Bookmarkable, shareable. Reload preserves period.
  • No time-of-day controls. Day-level granularity is correct for business ops.
Source: Looker Studio (with grouping pattern from Mixpanel/Amplitude) Resolves your "date and time settings are not intuitive" complaint directly. The rolling-vs-calendar distinction (Last 7 days vs This week) is the load-bearing UX detail — group them visually so the user sees the difference at-a-glance.

Research source: Looker Studio date range control docs; Amplitude/Mixpanel preset grouping conventions.

D3Kanban filter + sort locked research contradicts benchmark #

You named Trello as the standard. Research found Linear is actually best-in-class for an ops kanban. Trello's filter sidebar wastes real estate; Linear's chip bar is cleaner. Flagging so you can confirm or push back before the Setter Kanban rework.

Decision (pending your confirmation): Linear's filter chip bar pattern. Specifically:

  • Inline chip bar (not sidebar). Press F to open a command-style picker. Active filters render as chips in a persistent bar directly under the view header. Each chip shows dimension + value with a clickable ×. Bar disappears when zero filters active.
  • Multi-select OR within dimension, AND across dimensions. Sarah + Juan in Assignee = either. Adding Priority = Urgent narrows to those with both. Correct for sales kanban use case.
  • Search-within-cards is a SEPARATE text input, not part of the filter picker. Different interaction class.
  • Sort is a separate control from filters (one Sort button, separate location). Manual drag-to-reorder is global state — automatically disabled (greyed handles) when any sort is active.
  • Filter state persists in URL. Sales managers can share "Sarah's overdue follow-ups" as a link.
  • Empty state must be explicit: "No cards match your current filters" + "Clear filters" button. Don't show empty columns silently (Trello fails here).
Source: Linear (chip bar) + Notion (sort/drag interaction) This applies to Setter Kanban (the cited "horrible" surface), Closer Kanban, and Dialer Kanban. Standardize the same pattern across all three workstations.

Research source: Linear filters docs; Trello filter sidebar (anti-pattern); Notion sort/drag interaction.

§2Decisions pending your call

D4Visibility / tier rules pending #

Question: Three sub-decisions.

  1. Which tier(s) see each cockpit? Marketing cockpit = leadership? team? both? Same for sales, settings.
  2. Does the dictionary tab inherit the cockpit's auth tier, or have its own (e.g., always-leadership-visible even if the cockpit is team-readable)?
  3. Are some metrics tier-hidden? (e.g., cash collected hidden from non-leadership viewing the sales cockpit)
Recommendation: Marketing + sales cockpits = leadership-tier read; settings = super_admin write + leadership read. Dictionary tab = inherits cockpit tier (Brandon at leadership needs it). No tier-hidden metrics — if someone can see the cockpit, they see all tiles. Tier-hidden metrics create the same "what does this number mean" surprise we're trying to fix.

D5Source / weakness badge inline pending #

Question: Does every metric tile render an inline source-dot + weakness indicator? Or is that info dictionary-only (accessed via the (i) popover)?

Recommendation: Source dot inline (small colored dot matching SOURCE_COLORS from lib/design-tokens.ts) — gives at-a-glance "this comes from Stripe / GHL / Sheets". Weakness indicator NOT inline — surfaces in the (i) popover when present. Inline weakness would create visual noise on every tile.

D6Drill-down behavior pending #

Question: Clicking a KpiCard does what? Three options:

  • (a) Opens a modal with detail rows + breakdowns
  • (b) Filters the lead/EOD table below to that metric's population (active behavior on marketing today)
  • (c) Navigates to a dedicated metric-detail route
Recommendation: (b) for marketing (already implemented + working), (b) for sales (currently doesn't exist — add). (a) modal adds new surface; (c) navigates away from the cockpit context — both wrong. (b) keeps the drill-down inline and uses the existing detail table real estate.

D7Period comparison default pending #

Question: The pacing rows have a toggle: Goal / Last Mo / 3mo Avg. Which is the default view on first cockpit load?

Recommendation: Goal. The cockpit is for pacing toward a target. Last Mo and 3mo Avg are diagnostic ("am I trending up or down") — secondary. Goal answers "where am I vs where I need to be" — primary. Per [[feedback-usability-over-looking-cool]] the cockpit is operational, not analytical.

D8Pacing display selection pending #

Question: Which tiles get a pacing display, which form (PacingInline vs PacingBar), and which get no pacing at all?

Recommendation: All KPI tiles that have a target (marketing.spend, marketing.leads_booked, sales.cash_collected, sales.cash_contracted, sales.calls_scheduled, sales.close_rate, sales.net_show_rate) get pacing. Use PacingInline (compact, single row, more density) by default — matches your feedback-usability-over-looking-cool + project-comsat-wireframe-lean-mode direction. PacingBar reserved for the dedicated pacing panel if one exists (large hero pacing for headline cash metric).

D9Number-correctness contract pending #

Question: When data is stale, partially-fetched, or a source errored, what does the tile show?

  • (a) Last known value with a stale-indicator badge
  • (b) "—" or "loading" placeholder
  • (c) Last known value with the "as of [timestamp]" updated to show the stale time
  • (d) Silent: show old value, trust the cron
Recommendation: (c). Per rule-28 (no silent failures) the cockpit cannot show old data silently. The "as of" timestamp surfaces stale data naturally — if the cockpit shows "Cash Collected $41K (as of 14h ago)" the user knows the cron broke. Pair with Overseer alerting at the source — Telegram fires when the cron fails, the cockpit just reflects the truth.

D10Refresh cadence + "as of" indicator pending #

Question: Two sub-questions.

  1. Server-side refresh on page load only (current behavior), or auto-refresh every N minutes via polling/SWR?
  2. "Data as of" indicator — page-level (one timestamp for the whole cockpit) or per-source (separate timestamps for ad data, EOD data, payment data)?
Recommendation: (1) Auto-refresh every 5 minutes on the cockpit page, with a manual "refresh now" button. Current `AutoRefresh` wrapper (30 min) too slow for a live ops dashboard during business hours. (2) Per-source. One indicator per source group at the top of the cockpit (e.g., "Ads: 6h ago · EOD: live · Payments: live"). Looker Studio convention. Surfaces freshness per data class so users know which numbers to trust.
