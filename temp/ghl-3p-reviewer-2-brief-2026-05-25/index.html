Context What is this and why does it matter

HPFHQ has been running on GoHighLevel (GHL) for forms, booking, messaging, CRM-style pipelines, and webhook orchestration. We're replacing GHL with in-house equivalents inside HPFHQ for three reasons:

  1. Trust and verification. Multiple times in the past 6 months a metric or campaign was misattributed or stalled because of opaque GHL state. Emil's directive: "build everything dark, activate nothing until 3P-verified."
  2. Cost compounding. GHL fees scale poorly with HPF's growth; the in-house build is amortised against existing HPFHQ infrastructure.
  3. Strategic — knowledge surface. Owning the message + booking + form layer lets us feed events into HPFHQ's canonical knowledge surface (Nova, Pathfinder, Decision Log), which we can't with GHL.

The replacement is structured as 9 tracks across forms, booking, messaging, CRM/pipelines, monitoring, safety, sandbox, and final cutover. Codex (an AI reviewer) has run partial 3P passes on tracks 1 and 6. You are the second independent reviewer per the master plan's safety protocol — your pass catches what Codex's first pass missed.

Hard rule from the master plan (line 100): "The reviewer must not have built the code they're reviewing." Your CC scope is Medus/Bite integration onboarding — you have not built HPFHQ GHL code, so independence holds. Flag immediately if at any point you find yourself reviewing something you authored.

Thought What the architectural decisions look like and why

You'll be checking code against architectural intent. Here's the intent so you can audit against it, not just compare to the docs (which are partly stale).

1. Build dark, flip late

Code lands fully implemented but gated behind environment flags (e.g. BOOKING_ENGINE_MODE=live). No customer-facing behavior changes until the flag flips. Allows full pre-flip auditing in production environments without risking customer impact.

What you check: for every track, find the flag gate. Confirm the code path is reachable in production but inert. Confirm there's no accidental live path that bypasses the flag.

2. Two independent passes minimum

Codex's pass is "everything Codex notices." Your pass is "everything an independent human-led pass notices." Different review minds catch different classes of bug. Codex tends to find structural and obvious-pattern issues; humans tend to find domain-mismatch and stale-contract issues.

What you check: review Codex's audit at docs/plans/handoffs/ghl-overhaul-conductor-handover-2026-05-25.md AS A STARTING POINT, not as a checklist. Look at what Codex didn't look at.

3. Verified receipts before activation

The plan bans the word "certain" and demands "verified" (with receipt) or "untested." Every cutover step must produce a verification receipt — a log line, a screenshot, a SQL query result, a green Overseer probe. No flags flip on vibe.

What you check: for each thing the docs claim is "done" or "verified," confirm the receipt exists and matches.

4. Rollback ready, every step

Every flag flip has a rollback procedure documented. PITR (Point-In-Time Recovery) on Supabase is the backstop for data; per-route flag flip is the backstop for code paths.

What you check: for each cutover step, confirm the rollback procedure is documented AND testable. "Flip flag back to off" is acceptable; "manually fix data" is not.

5. Stale doc warning

docs/architecture/booking.md currently claims things that the code doesn't actually do. Specifically: it claims manage-token verification is timing-safe with 30-day expiry; the code doesn't yet implement timing-safe compare. It claims pre-call dedup uses message_sends; the code queries a non-existent message_log table.

What you check: for every claim in docs/architecture/booking.md, verify against actual code. Don't trust the doc.
